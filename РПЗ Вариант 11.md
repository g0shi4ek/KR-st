## 1. ВВЕДЕНИЕ

Настоящая расчётно-пояснительная записка описывает разработку программного обеспечения для организации передачи текстовых файлов между двумя персональными компьютерами, соединёнными через последовательный интерфейс RS232C нульмодемным кабелем.

Целью курсовой работы является практическое освоение принципов построения протоколов взаимодействия в локальных сетях на нескольких уровнях модели OSI: физическом, канальном и прикладном. В рамках работы реализованы:

- физическое соединение двух ПК через интерфейс RS232C нульмодемным кабелем;
- канальный уровень с кадрированием данных, подтверждением доставки и повторной передачей при ошибках (протокол ARQ «стоп-и-жди»);
- помехозащитное кодирование передаваемых данных кодом Хэмминга [7,4] с исправлением одиночных ошибок;
- прикладной уровень с функцией передачи текстовых файлов, просмотром содержимого файла в реальном времени на стороне получателя и управлением темпом передачи.

Программный продукт реализован на языке **C#** с использованием технологии **Windows Forms** (.NET 8) и библиотеки `System.IO.Ports` для работы с последовательным портом.

## 2. ТРЕБОВАНИЯ К ПРОГРАММЕ

### 2.1 Функциональные требования

Программа должна обеспечивать:

1. **Выбор роли** при запуске: «Отправитель» (Sender) или «Получатель» (Receiver).
2. **Установление логического соединения** между двумя ПК через COM-порт (COM1 для отправителя, COM2 для получателя) по нульмодемному кабелю.
3. **Передачу текстового файла** от отправителя к получателю:
   - отправитель выбирает файл через диалог открытия файла;
   - файл разбивается на блоки (чанки) по 128 байт;
   - каждый блок кодируется кодом Хэмминга [7,4] перед отправкой;
   - получатель декодирует каждый блок с исправлением одиночных ошибок.
4. **Просмотр содержимого файла** на стороне получателя в реальном времени по мере поступления блоков.
5. **Управление темпом передачи** на прикладном уровне:
   - три уровня темпа: Медленный (500 мс), Нормальный (100 мс), Быстрый (10 мс) — задержка между отправкой блоков;
   - как отправитель, так и получатель могут изменить темп в процессе передачи.
6. **Сохранение принятого файла** на диск по завершении передачи.
7. **Журналирование** всех событий (отправка/приём кадров, ошибки, смена темпа) в текстовом поле.
8. **Разрыв логического соединения** по инициативе любой из сторон.

## 4. ФИЗИЧЕСКИЙ УРОВЕНЬ

### 4.1 Функции физического уровня

Физический уровень обеспечивает:

- **Механическое соединение**: разъёмы DB-9 (9-контактный) на обоих ПК.
- **Электрические характеристики**: уровни сигналов RS232C (логический «0» — от +3 до +15 В, логическая «1» — от −3 до −15 В).
- **Функциональные характеристики**: назначение контактов (TxD, RxD, RTS, CTS, DTR, DSR, GND).
- **Процедурные характеристики**: последовательная передача бит, асинхронный режим (старт-бит, 8 бит данных, стоп-бит).

### 4.2 Описание интерфейса RS232C

Интерфейс RS232C (EIA-232) — стандарт последовательной передачи данных между двумя устройствами (DTE — Data Terminal Equipment и DCE — Data Communications Equipment).

| Параметр | Значение |
|---|---|
| Максимальная скорость | до 115 200 бит/с |
| Максимальная длина кабеля | ~15 м при 9600 бит/с |
| Уровень логического «0» | +3 … +15 В |
| Уровень логической «1» | −3 … −15 В |
| Режим передачи | Асинхронный, дуплексный |
| Разъём | DB-9 |

**Назначение контактов DB-9:**

| Контакт | Сигнал | Направление | Описание |
|---|---|---|---|
| 1 | DCD | Вход | Data Carrier Detect |
| 2 | RxD | Вход | Receive Data |
| 3 | TxD | Выход | Transmit Data |
| 4 | DTR | Выход | Data Terminal Ready |
| 5 | GND | — | Signal Ground |
| 6 | DSR | Вход | Data Set Ready |
| 7 | RTS | Выход | Request To Send |
| 8 | CTS | Вход | Clear To Send |
| 9 | RI | Вход | Ring Indicator |

**Формат асинхронного символа (8N1):**

```
  Линия: ─────┐   ┌───┬───┬───┬───┬───┬───┬───┬───┐   ┌─────
              │   │D0 │D1 │D2 │D3 │D4 │D5 │D6 │D7 │   │
              └───┘   └───┴───┴───┴───┴───┴───┴───┘   │
              Старт   ←──── 8 бит данных ────────►   Стоп
               (0)                                     (1)
```

При скорости 9600 бит/с длительность одного бита ≈ 104 мкс. Один символ (10 бит) передаётся за ~1,04 мс. Максимальная пропускная способность: 9600 / 10 = **960 байт/с**.

### 4.3 Нуль-модемный интерфейс

Нуль-модемный кабель позволяет соединить два ПК напрямую без модемов, перекрещивая линии передачи и приёма, а также линии управления потоком.

**Схема нуль-модемного кабеля (DB-9 ↔ DB-9):**

```
  ПК-1 (COM1)                              ПК-2 (COM2)
  ─────────────────────────────────────────────────────
  TxD  (3) ──────────────────────────────── RxD  (2)
  RxD  (2) ──────────────────────────────── TxD  (3)
  RTS  (7) ──────────────────────────────── CTS  (8)
  CTS  (8) ──────────────────────────────── RTS  (7)
  DTR  (4) ──────────────────────────────── DSR  (6)
  DSR  (6) ──────────────────────────────── DTR  (4)
  GND  (5) ──────────────────────────────── GND  (5)
```

Данные, передаваемые по линии TxD одного ПК, принимаются по линии RxD другого ПК, и наоборот. Сигналы RTS/CTS и DTR/DSR перекрещиваются, что позволяет каждому ПК «видеть» готовность партнёра.

В данной работе оба сигнала RTS и DTR устанавливаются в `true` при открытии порта, что обеспечивает постоянную готовность к обмену без аппаратного управления потоком.

## 5. НАСТРОЙКА COM-ПОРТА СРЕДСТВАМИ C#

### 5.1 Класс SerialPort

Для работы с последовательным портом в C# используется класс `System.IO.Ports.SerialPort` из пакета NuGet `System.IO.Ports`.

**Основные свойства:**

| Свойство | Тип | Описание |
|---|---|---|
| `PortName` | `string` | Имя порта (например, "COM1") |
| `BaudRate` | `int` | Скорость передачи (бит/с) |
| `DataBits` | `int` | Количество бит данных (5–8) |
| `Parity` | `Parity` | Чётность (None, Odd, Even, Mark, Space) |
| `StopBits` | `StopBits` | Стоп-биты (One, OnePointFive, Two) |
| `RtsEnable` | `bool` | Управление сигналом RTS |
| `DtrEnable` | `bool` | Управление сигналом DTR |
| `ReadTimeout` | `int` | Таймаут чтения (мс) |
| `WriteTimeout` | `int` | Таймаут записи (мс) |
| `BytesToRead` | `int` | Количество байт в буфере приёма |

**Основные методы:**

| Метод | Описание |
|---|---|
| `Open()` | Открыть порт |
| `Close()` | Закрыть порт |
| `Write(byte[], int, int)` | Записать массив байт |
| `ReadByte()` | Прочитать один байт (блокирующий) |
| `Read(byte[], int, int)` | Прочитать массив байт |

**События:**

| Событие | Описание |
|---|---|
| `DataReceived` | Вызывается при поступлении данных в буфер |
| `ErrorReceived` | Вызывается при ошибке порта |

Событие `DataReceived` вызывается в отдельном потоке. Для обновления UI из этого потока используется `Control.Invoke()` / `Control.BeginInvoke()`.

### 5.2 Класс PortSetup

В данной работе настройка COM-порта вынесена в статический класс `PortSetup` (файл `PortSetup.cs`). Поскольку по условию задания параметры порта заданы по умолчанию, класс содержит только константы и один метод открытия порта. Диалог настроек не предусмотрен.

**Константы класса `PortSetup`:**

| Константа | Значение | Описание |
|---|---|---|
| `SenderPortName` | `"COM1"` | Порт отправителя |
| `ReceiverPortName` | `"COM2"` | Порт получателя |
| `DefaultBaudRate` | `9600` | Скорость передачи |
| `DefaultParity` | `Parity.None` | Чётность |
| `DefaultDataBits` | `8` | Биты данных |
| `DefaultStopBits` | `StopBits.One` | Стоп-биты |

**Метод `OpenPort(string portName)`:**

```csharp
public static SerialPort OpenPort(string portName)
{
    var port = new SerialPort(portName)
    {
        BaudRate     = DefaultBaudRate,
        Parity       = DefaultParity,
        DataBits     = DefaultDataBits,
        StopBits     = DefaultStopBits,
        RtsEnable    = true,
        DtrEnable    = true,
        ReadTimeout  = 3000,
        WriteTimeout = 3000
    };
    port.Open();
    return port;
}
```

Алгоритм открытия COM-порта приведён на **Листе 6, схема 6а** (файл «Схемы для РПЗ (Mermaid).md»).

## 6. КАНАЛЬНЫЙ УРОВЕНЬ

### 6.1 Функции канального уровня

Канальный уровень обеспечивает надёжную передачу данных между двумя узлами через физический канал. В данной работе реализованы следующие функции:

1. **Кадрирование** — данные разбиваются на кадры фиксированной структуры с маркером начала `0xFF` (11111111) и байтом типа кадра.
2. **Адресация** — в данной топологии (2 ПК) адресация не требуется; роли определяются при запуске.
3. **Обнаружение и исправление ошибок** — данные файла защищаются кодом Хэмминга [7,4] с исправлением одиночных ошибок.
4. **Управление потоком** — механизм «стоп-и-жди» (Stop-and-Wait ARQ): каждый кадр данных ожидает подтверждения (ACK) перед отправкой следующего.
5. **Повторная передача** — при получении кадра RET отправитель повторяет последний блок.
6. **Установление и разрыв логического соединения** — кадры UPLINK/ACK_UPLINK/DOWNLINK/ACK_DOWNLINK.

### 6.2 Протокол канального уровня

Протокол реализован в классе `Frames` (файл `Frames.cs`).

Структурные схемы процедур взаимодействия приведены на **Листе 1**, временные диаграммы — на **Листе 2** (файл «Схемы для РПЗ (Mermaid).md»).

#### 6.2.1 Установление логического соединения

```
  ПК-1 (Отправитель, COM1)        ПК-2 (Получатель, COM2)
        │                               │
        │──── UPLINK [0xFF][0x01] ─────►│
        │◄─── ACK_UPLINK [0xFF][0x02] ──│
        │                               │
        │    (соединение установлено)   │
```

При неуспешном установлении (нет ответа) отправитель повторяет UPLINK-кадр до 3 раз с таймаутом 3000 мс. После 3 неудачных попыток выводится сообщение об ошибке.

Алгоритм установления соединения приведён на **Листе 6, схема 6в**.

#### 6.2.2 Передача файла

```
  ПК-1 (Отправитель)              ПК-2 (Получатель)
        │                               │
        │── FILE_INFO [0xFF][0x10] ────►│  имя файла + кол-во блоков
        │◄── ACK_FILE_INFO [0xFF][0x11] │
        │                               │
        │── CHUNK[0] [0xFF][0x20] ─────►│  блок 0, закодирован Хэммингом
        │◄── ACK_CHUNK [0xFF][0x21] ────│
        │                               │
        │── CHUNK[1] [0xFF][0x20] ─────►│
        │◄── ACK_CHUNK [0xFF][0x21] ────│
        │         ...                   │
        │── CHUNK[N-1] [0xFF][0x20] ───►│
        │◄── ACK_CHUNK [0xFF][0x21] ────│
        │                               │
        │── TRANSFER_DONE [0xFF][0x40] ►│
        │◄── ACK_DONE [0xFF][0x41] ─────│
```

Алгоритм отправки файла приведён на **Листе 6, схема 6з**.

#### 6.2.3 Повторная передача при ошибке

```
  ПК-1 (Отправитель)              ПК-2 (Получатель)
        │                               │
        │── CHUNK[N] [0xFF][0x20] ─────►│  ошибка декодирования Хэмминга
        │◄── RET_CHUNK [0xFF][0x22] ────│  запрос повтора
        │                               │
        │── CHUNK[N] [0xFF][0x20] ─────►│  повторная отправка
        │◄── ACK_CHUNK [0xFF][0x21] ────│  успешно принято
```

#### 6.2.4 Управление темпом передачи

```
  ПК-2 (Получатель)               ПК-1 (Отправитель)
        │                               │
        │── PACE_SLOW [0xFF][0x30] ────►│  задержка 500 мс
        │── PACE_NORMAL [0xFF][0x31] ──►│  задержка 100 мс
        │── PACE_FAST [0xFF][0x32] ────►│  задержка  10 мс
```

При получении кадра темпа отправитель немедленно применяет новую задержку (volatile-поле `_currentPaceMs` в `ChatForm`).

#### 6.2.5 Разрыв логического соединения

```
  ПК-1 (Отправитель)              ПК-2 (Получатель)
        │                               │
        │── DOWNLINK [0xFF][0x04] ─────►│
        │◄── ACK_DOWNLINK [0xFF][0x05] ─│
        │                               │
        │    (соединение разорвано)     │
```

#### 6.2.6 Синхронизация ACK (ManualResetEventSlim)

Метод `SendChunk` отправляет кадр CHUNK и блокируется на объекте `ManualResetEventSlim _chunkAckEvent` с таймаутом 5 секунд. Когда обработчик `DataReceived` получает ACK_CHUNK или RET_CHUNK, он вызывает `FrameAction`, который устанавливает сигнал (`_chunkAckEvent.Set()`). Это исключает конфликт между потоком отправки и обработчиком `DataReceived`.

```
  Поток SendFileThread              Поток DataReceived
        │                                    │
        │  SendChunk(N, data, port)           │
        │  ──► Write CHUNK frame              │
        │  ──► _chunkAckEvent.Reset()         │
        │  ──► _chunkAckEvent.Wait(5000) ─────┤ (блокировка)
        │                          FrameAction(ACK_CHUNK)
        │                          _chunkAckEvent.Set()
        │◄────────────────────────────────────┤
        │  (разблокировка, продолжение)       │
```

### 6.3 Защита информации кодом Хэмминга [7,4]

#### 6.3.1 Теоретические основы

Код Хэмминга [7,4] — линейный блочный код с параметрами:

| Параметр | Значение |
|---|---|
| n | 7 — длина кодового слова (бит) |
| k | 4 — количество информационных бит |
| r | 3 — количество проверочных бит (r = n − k) |
| d_min | 3 — минимальное кодовое расстояние |
| Исправляет | одиночные ошибки |
| Обнаруживает | двойные ошибки |

Условие: 2^r ≥ n + 1, т.е. 2³ = 8 ≥ 8 ✓.

#### 6.3.2 Структура кодового слова

```
  Позиция:  1    2    3    4    5    6    7
            p₁   p₂   d₁   p₃   d₂   d₃   d₄
```

Проверочные биты контролируют:

```
  p₁ (поз. 1): позиции 1, 3, 5, 7  →  p₁ = d₁ ⊕ d₂ ⊕ d₄
  p₂ (поз. 2): позиции 2, 3, 6, 7  →  p₂ = d₁ ⊕ d₃ ⊕ d₄
  p₃ (поз. 4): позиции 4, 5, 6, 7  →  p₃ = d₂ ⊕ d₃ ⊕ d₄
```

#### 6.3.3 Кодирование (метод `EncodeNibble`)

В реализации (класс `Coding`, метод `EncodeNibble`) биты нумеруются с 0:

```csharp
// d[0..3] — информационные биты (d₁..d₄)
// c[0..6] — кодовое слово (7 бит)
c[0] = d[0] ^ d[1] ^ d[3];  // p₁
c[1] = d[0] ^ d[2] ^ d[3];  // p₂
c[2] = d[1] ^ d[2] ^ d[3];  // p₃
c[3] = d[0];                 // d₁
c[4] = d[1];                 // d₂
c[5] = d[2];                 // d₃
c[6] = d[3];                 // d₄
```

Алгоритм кодирования нибла приведён на **Листе 6, схема 6е**.

**Пример кодирования нибла `1011` (d₁=1, d₂=0, d₃=1, d₄=1):**

```
  p₁ = 1 ⊕ 0 ⊕ 1 = 0
  p₂ = 1 ⊕ 1 ⊕ 1 = 1
  p₃ = 0 ⊕ 1 ⊕ 1 = 0
  Кодовое слово: 0 1 0 1 0 1 1  (p₁ p₂ p₃ d₁ d₂ d₃ d₄)
```

#### 6.3.4 Декодирование и исправление ошибок (метод `DecodeNibble`)

При декодировании вычисляется **синдром** — трёхбитовое число, указывающее на позицию ошибки (или 0, если ошибок нет):

```csharp
s0 = c[6] ^ c[4] ^ c[2] ^ c[0];  // проверка по p₁
s1 = c[5] ^ c[4] ^ c[1] ^ c[0];  // проверка по p₂
s2 = c[3] ^ c[2] ^ c[1] ^ c[0];  // проверка по p₃

// Позиция ошибки (1-based)
int errPos = (s0 ? 4 : 0) + (s1 ? 2 : 0) + (s2 ? 1 : 0);
// Если errPos != 0 — инвертировать бит c[errPos-1]
```

Алгоритм декодирования приведён на **Листе 6, схема 6ж**.

**Таблица синдромов:**

| s0 | s1 | s2 | errPos | Позиция ошибки |
|---|---|---|---|---|
| 0 | 0 | 0 | 0 | Ошибок нет |
| 0 | 0 | 1 | 1 | c[0] (p₁) |
| 0 | 1 | 0 | 2 | c[1] (p₂) |
| 0 | 1 | 1 | 3 | c[2] (p₃) |
| 1 | 0 | 0 | 4 | c[3] (d₁) |
| 1 | 0 | 1 | 5 | c[4] (d₂) |
| 1 | 1 | 0 | 6 | c[5] (d₃) |
| 1 | 1 | 1 | 7 | c[6] (d₄) |

#### 6.3.5 Кодирование байтов файла

Каждый байт данных разбивается на два нибла:
- **Младший нибл** (биты 0–3) кодируется в 7 бит → 1 байт;
- **Старший нибл** (биты 4–7) кодируется в 7 бит → 1 байт.

**Соотношение размеров:**

| Исходные данные | После кодирования |
|---|---|
| 1 байт (8 бит) | 2 байта (14 информационных бит) |
| 128 байт (блок) | 256 байт (закодированный блок) |

Методы класса `Coding`:

| Метод | Описание |
|---|---|
| `EncodeNibble(BitArray d)` | Кодирует 4-битный нибл в 7-битное кодовое слово |
| `DecodeNibble(BitArray c)` | Декодирует 7-битное слово, исправляет одиночную ошибку |
| `EncodeBytes(byte[])` | Кодирует массив байт (для данных файла) |
| `DecodeBytes(byte[])` | Декодирует массив байт с исправлением ошибок |
| `Encode(string)` | Кодирует строку (с конвертацией в Win-1251) |
| `Decode(byte[])` | Декодирует байты в строку |

### 6.4 Формат кадров

Все кадры имеют единый формат начала: маркер `0xFF` (11111111) + байт типа кадра. Структуры PDU приведены на **Листе 3** (файл «Схемы для РПЗ (Mermaid).md»).

**Служебные кадры (2 байта):**

```
┌──────────┬──────────┐
│ 11111111 │ ТипКадра │
│  1 байт  │  1 байт  │
└──────────┴──────────┘
```

**Кадр FILE_INFO (переменная длина):**

```
┌──────────┬──────────┬──────────────────┬──────────────────────┬──────────────────┐
│ 11111111 │ 00010000 │  name_len        │  Имя файла           │  total_chunks    │
│  1 байт  │  1 байт  │  2 байта (LE)   │  name_len байт       │  4 байта (LE)    │
└──────────┴──────────┴──────────────────┴──────────────────────┴──────────────────┘
```

**Кадр CHUNK (переменная длина):**

```
┌──────────┬──────────┬──────────────────┬──────────────────┬──────────────────────────┐
│ 11111111 │ 00100000 │  chunk_index     │  data_len        │  Данные (Хэмминг [7,4])  │
│  1 байт  │  1 байт  │  4 байта (LE)   │  2 байта (LE)    │  до 256 байт             │
└──────────┴──────────┴──────────────────┴──────────────────┴──────────────────────────┘
```

**Таблица всех типов кадров:**

| Тип кадра | Код (hex) | Код (bin) | Направление | Описание |
|---|---|---|---|---|
| UPLINK | 0x01 | 00000001 | S → R | Запрос установления соединения |
| ACK_UPLINK | 0x02 | 00000010 | R → S | Подтверждение соединения |
| RET_UPLINK | 0x03 | 00000011 | R → S | Отказ в соединении |
| DOWNLINK | 0x04 | 00000100 | S → R | Запрос разрыва соединения |
| ACK_DOWNLINK | 0x05 | 00000101 | R → S | Подтверждение разрыва |
| FILE_INFO | 0x10 | 00010000 | S → R | Метаданные файла |
| ACK_FILE_INFO | 0x11 | 00010001 | R → S | Подтверждение метаданных |
| RET_FILE_INFO | 0x12 | 00010010 | R → S | Запрос повтора метаданных |
| CHUNK | 0x20 | 00100000 | S → R | Блок данных файла (Хэмминг) |
| ACK_CHUNK | 0x21 | 00100001 | R → S | Подтверждение блока |
| RET_CHUNK | 0x22 | 00100010 | R → S | Запрос повтора блока |
| PACE_SLOW | 0x30 | 00110000 | S ↔ R | Установить темп: 500 мс |
| PACE_NORMAL | 0x31 | 00110001 | S ↔ R | Установить темп: 100 мс |
| PACE_FAST | 0x32 | 00110010 | S ↔ R | Установить темп: 10 мс |
| TRANSFER_DONE | 0x40 | 01000000 | S → R | Передача завершена |
| ACK_DONE | 0x41 | 01000001 | R → S | Подтверждение завершения |

> Стартовый байт для всех кадров: **0xFF (11111111)**

## 7. ПРИКЛАДНОЙ УРОВЕНЬ

### 7.1 Функции прикладного уровня

Прикладной уровень реализован в классе `ChatForm` (файл `Forms/ChatForm.cs`) и обеспечивает:

1. **Выбор роли** — при запуске отображается `LoginForm` с кнопками «Отправитель» и «Получатель».
2. **Управление соединением** — кнопки «Подключиться» / «Отключиться».
3. **Выбор и отправку файла** — кнопка «Обзор» открывает диалог выбора `.txt` файла; кнопка «Отправить файл» запускает фоновый поток `SendFileThread`.
4. **Просмотр содержимого файла** — по мере поступления блоков CHUNK получатель отображает декодированный текст в поле `rtbFileContent`.
5. **Управление темпом** — кнопки «Медленно» / «Нормально» / «Быстро» отправляют кадры PACE_SLOW / PACE_NORMAL / PACE_FAST.
6. **Сохранение файла** — кнопка «Сохранить файл» собирает блоки в порядке `chunk_index` и записывает на диск.
7. **Журналирование** — все события записываются в `rtbLog` с цветовой маркировкой.

### 7.2 Форма выбора роли (LoginForm)

При запуске программы отображается `LoginForm` с тремя кнопками:
- **«Отправитель»** — устанавливает `IsSender = true`, закрывает форму.
- **«Получатель»** — устанавливает `IsSender = false`, закрывает форму.
- **«Выход»** — завершает приложение.

Граф диалога пользователя приведён на **Листе 5** (файл «Схемы для РПЗ (Mermaid).md»).

### 7.3 Главная форма (ChatForm)

Главная форма содержит следующие группы элементов управления:

| Группа | Элементы | Описание |
|---|---|---|
| `grpConnection` | Метка роли, метка порта, статус, кнопки «Подключиться» / «Отключиться» | Управление соединением |
| `grpSender` | Поле пути файла, кнопка «Обзор», кнопка «Отправить файл», прогресс-бар, кнопки темпа | Только для отправителя |
| `grpReceiver` | Метка имени файла, кнопка «Сохранить файл», прогресс-бар, кнопки темпа | Только для получателя |
| `grpFilePreview` | `rtbFileContent` (RichTextBox) | Просмотр содержимого файла |
| `grpLog` | `rtbLog` (RichTextBox) | Журнал событий |

Группа `grpSender` видима только для отправителя, `grpReceiver` — только для получателя.

### 7.4 Процедура отправки файла

1. Пользователь нажимает «Обзор» → выбирает `.txt` файл.
2. Нажимает «Отправить файл» → запускается фоновый поток `SendFileThread`.
3. Поток читает файл, разбивает на блоки по 128 байт, отправляет `FILE_INFO`.
4. Для каждого блока: кодирует Хэммингом → отправляет `CHUNK` → ждёт `ACK_CHUNK` → делает паузу темпа.
5. При получении `RET_CHUNK` — повторяет блок.
6. После всех блоков отправляет `TRANSFER_DONE`, ждёт `ACK_DONE`.

Алгоритм отправки файла приведён на **Листе 6, схема 6з**.

### 7.5 Процедура приёма файла

1. При получении `FILE_INFO` — отображает имя файла, инициализирует массив блоков.
2. При получении каждого `CHUNK` — декодирует Хэмминг, сохраняет блок по индексу, обновляет просмотр файла и прогресс-бар.
3. При ошибке декодирования — отправляет `RET_CHUNK`.
4. При получении `TRANSFER_DONE` — отправляет `ACK_DONE`, активирует кнопку «Сохранить файл».
5. Пользователь нажимает «Сохранить файл» → блоки собираются в порядке `chunk_index` → записываются на диск.

Алгоритм приёма кадров приведён на **Листе 6, схема 6г**.

### 7.6 Управление темпом передачи

Темп передачи управляется на прикладном уровне. Задержка применяется в потоке `SendFileThread` между отправкой блоков:

| Уровень темпа | Задержка | Кадр |
|---|---|---|
| Медленный (Slow) | 500 мс | PACE_SLOW (0x30) |
| Нормальный (Normal) | 100 мс | PACE_NORMAL (0x31) |
| Быстрый (Fast) | 10 мс | PACE_FAST (0x32) |

Текущий темп хранится в `volatile int _currentPaceMs` в `ChatForm`. При получении кадра темпа вызывается `HandlePaceChanged`, который обновляет это поле. Поток `SendFileThread` читает его перед каждой паузой. Как отправитель, так и получатель могут изменить темп в любой момент передачи.
